GitとGitHubってなに？

まずGitってなーに？

Gitを一言で表すとバージョン管理システムです

つまり，Gitを使ってファイルの保存などが行えれます

Gitを使ってファイルの保存を行うと，誰が，いつ，どこを，どのように変更したのかが記録されます

その履歴を使って，今変更した箇所をすべて帳消しにしたり，ある履歴で変更した箇所だけを削除したり，ある履歴まで戻ることができます

この機能は過去に追加した機能を削除したい，戻りたい場合にも便利ですが，一番便利だと思っている点はちょっとだけテストで実装したいけど，だめだったら削除したいという場合だと思います

たくさんの変更をテストで実装して，それを削除したい場合に変更した場所を一生懸命に探してバックスペースキーを連打しなくてもコマンド一発で過去の履歴からの変更点を消してくれます

マリオの中間ポイント的な感じです


他のメリットとして，マージが簡単にできる点です

ファイルは通常上書き保存な性質を持っています

例えば同じ名前のtest.txtファイルを2人で実装する場合に，同じディレクトリに保存をしてしまうと一番最後に保存したファイルが上書き保存により残ってしまうため，はじめの人の変更点は反映されないことになってしまいます

また，保存してもらうディレクトリを工夫しても，マージする作業者はとても大変な作業になります

一つ一つのファイルを目視で確認し，変更された箇所を抜き取り，一つファイルにまとめ上げるのです

そのようなことを上手に管理してくれるのがGitなのです

複数人で1つのファイルを編集している場合でも，Gitは差分を読み取り，自動的にマージ作業を行ってくれます

ですが，最終的な決定権は人間にあります

Gitが差分を取ってマージをしようとしたが，どちらのファイルの変更が正しいのかがわからなくなった場合にはコンフリクト(日本語訳では衝突)が発生します

コンフリクトが発生してしまった場合，私達人間がどちらが正しいのかを判断しなければなりません

じゃあGitHubってなーに？

よくわからない人はおそらくGitとGtiHubは同じものだと思っている人も多いかもしれません

Gtiはバージョン管理システムであり，ファイルを保存するソフトウェアですが，実はGit自体には他の人とファイルを共有する仕組みはありますが方法がありません

そこで使われるのがGitHubです

Gitで作った履歴をGitHubにアップロードして，様々な人とファイルを共有します

そしてその共有した歴史を他の人がダウンロードして，ダウンロードした歴史をもとに実装などをして変更した歴史を再びGitHubへアップロードして，GitHub上でファイルをマージします

もっと簡単に言うと，他人とファイルを共有するために僕たちが使うGoogle DriveやOnedrive,DropBoxと言ったクラウドサービスのGitバージョンと言った感じです

まとめ

  • Gitは歴史を作るバージョン管理システム！
  • GitHubはGitで作った歴史を他の人と共有するクラウドサービス！
